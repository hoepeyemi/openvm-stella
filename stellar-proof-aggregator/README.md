# OpenVM × Stellar: Aggregated Proof-of-Balance

> **Hackathon project** — Proof Aggregation / Recursive Proofs on Stellar
>
> Collapse N individual ZK proof verifications into a single Stellar transaction
> using OpenVM's STARK aggregation and BN254 pairing host functions.

## What this does

```
N users each hold a secret balance
      ↓
N × SHA-256 commitment + range-check proofs
      ↓  (inside OpenVM zkVM, off-chain)
ONE aggregate STARK proof  →  Groth16 SNARK (BN254)
      ↓  (single Stellar transaction)
Soroban contract: pairing_check(4 pairs) → ✓
```

**Instead of N separate on-chain verifications (each costing ~27.5M Soroban instructions),
a single aggregated proof costs ~29M instructions regardless of N.**

| N  | Without aggregation | With aggregation | Savings |
|----|--------------------:|----------------:|--------:|
|  5 |        137,500,000 |      29,000,000 |   78.9% |
| 10 |        275,000,000 |      29,000,000 |   89.5% |
| 50 |      1,375,000,000 |      29,000,000 |   97.9% |

## Architecture

```
stellar-proof-aggregator/
├── guest/balance-aggregator/    # OpenVM RISC-V guest program
│   ├── src/main.rs              # Verifies N commitments + range checks inside zkVM
│   └── openvm.toml              # rv32i + rv32m + io extensions
│
├── host/                        # Host-side proof orchestration (std Rust)
│   └── src/
│       ├── main.rs              # Build ELF → execute → prove → benchmark
│       ├── proof_types.rs       # BalanceWitness, AggregatedPublicInputs
│       └── stellar_export.rs    # Cost model & Stellar submission formatter
│
└── contracts/stellar-verifier/  # Soroban smart contract
    └── src/lib.rs               # Groth16 BN254 verifier using Stellar host functions
```

## Inner proof type: Proof-of-Balance

Each "inner proof" is a commitment-and-range statement:

- **Private witness**: `(balance: u64, nonce: u64)`
- **Public commitment**: `SHA-256(balance_le8 || nonce_le8)`
- **Statement**: `0 ≤ balance ≤ 1,000,000,000,000` AND `commitment == SHA256(balance || nonce)`

The OpenVM guest program verifies all N such statements simultaneously.  OpenVM
produces a single STARK proof covering all N verifications, which is then
compressed to a Groth16 BN254 SNARK for on-chain verification.

## Quick start

### Prerequisites

```bash
# OpenVM toolchain (Rust 1.90, nightly for guest)
rustup toolchain install nightly-2025-08-02
rustup component add rust-src --toolchain nightly-2025-08-02
cargo install cargo-nextest

# Stellar CLI (for contract deployment)
cargo install --locked stellar-cli --features opt
```

### 1. Build the guest program

```bash
cd guest/balance-aggregator
cargo build --release
```

### 2. Run the host orchestrator (generates STARK proof + benchmark)

```bash
cd ../../host
cargo run --release -- --n 10
```

This:
1. Compiles the guest to RISC-V ELF via OpenVM toolchain
2. Generates 10 random balance witnesses
3. Executes the guest (fast dry-run)
4. Generates an aggregate STARK proof (takes a few minutes)
5. Verifies the STARK proof locally
6. Prints Stellar cost comparison

### 3. Build the Soroban verifier contract

```bash
cd ../contracts/stellar-verifier
cargo build --target wasm32-unknown-unknown --release
```

### 4. Deploy to Stellar Testnet

```bash
stellar contract deploy \
  --wasm target/wasm32-unknown-unknown/release/stellar_aggregated_verifier.wasm \
  --source <your-keypair> \
  --network testnet

# Initialize with the verifying key (generated by OpenVM keygen)
stellar contract invoke \
  --id <CONTRACT_ID> \
  --source <your-keypair> \
  --network testnet \
  -- initialize \
  --admin <ADMIN_ADDRESS> \
  --vk "$(cat vk.json)"
```

### 5. Submit an aggregated proof

```bash
stellar contract invoke \
  --id <CONTRACT_ID> \
  --source <user-keypair> \
  --network testnet \
  -- verify_aggregate \
  --proof '{"a":{"x":"...","y":"..."},"b":...,"c":...}' \
  --pub_inputs '[n_proofs_as_fr, hash_chunk_0, ..., hash_chunk_7]'
```

## Proof pipeline (production)

```
OpenVM STARK (BabyBear)
   → Leaf STARK (recursive)
   → Internal STARK (recursive)
   → Root STARK
   → Groth16 BN254 SNARK  ← generated via ark-groth16 or snark-verifier wrapper
         ↓
   Soroban AggregatedVerifier.verify_aggregate(proof, vk, pub_inputs)
         ↓
   env.crypto().bn254().pairing_check([−A, α, vk_x, C], [B, β, γ, δ]) == true
```

The Groth16 wrapper around the OpenVM root STARK is the same cryptographic
step that OpenVM uses for EVM verification (via Halo2 KZG), adapted to use
Groth16 so the Soroban contract only needs the three pairing host functions
available since Protocol 25.

## Stellar host functions used

| Function | Protocol | Purpose |
|----------|----------|---------|
| `env.crypto().bn254().g1_mul(point, scalar)` | 25 | Compute vk_x terms |
| `env.crypto().bn254().g1_add(p1, p2)` | 25 | Accumulate vk_x |
| `env.crypto().bn254().pairing_check(g1s, g2s)` | 25 | Final Groth16 check |

Protocol 26 adds `g1_msm` (multi-scalar multiplication), which would let us
replace the vk_x loop with a single call — further reducing instruction cost.

## Security notes

- The verifying key is circuit-specific and must be generated from a trusted setup.
- The commitment scheme is based on SHA-256, which is sound but not hiding against
  birthday attacks on the nonce space; for production use Poseidon2 with a larger nonce.
- Replay protection: the contract counts verified batches but does not enforce
  nullifiers — add a nullifier set for applications requiring unique-use proofs.
